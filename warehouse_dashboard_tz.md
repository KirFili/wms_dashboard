# Техническое задание: web-дашборд «Управление складом»

## 1. Назначение проекта

Разработать web-приложение класса dashboard / internal web app для мониторинга и анализа использования паллетомест на складе, управления справочниками и загрузки данных из 1С в формате XLSX. Интерфейс должен поддерживать оперативный контроль текущей заполненности склада, анализ динамики по выбранному периоду, оценку эффективности хранения по SKU и администрирование структуры камер хранения.[cite:5][cite:13]

Система должна строиться как data-dense web application: с боковой навигацией, единым основным scroll-region, верхней зоной глобальных фильтров, блоком KPI, аналитическими графиками и детальными таблицами. Для аналитических интерфейсов такой паттерн считается базовым, так как помогает быстро переходить от агрегированных показателей к детализации.[cite:4]

## 2. Бизнес-цели

Основные бизнес-цели проекта:

- дать складским менеджерам прозрачную картину занятости и свободных паллетомест по камерам хранения;
- показать, какая номенклатура занимает склад сейчас и как менялась занятость по периоду;
- выявлять SKU с низкой эффективностью использования склада через сопоставление объемов хранения и отгрузок;
- сократить время на подготовку аналитики за счет регулярной загрузки XLSX-файлов из 1С;
- централизовать справочники камер, паллетомест и SKU в одной системе.[cite:2][cite:3][cite:13]

## 3. Пользователи и роли

### 3.1 Роли

| Роль | Полномочия |
|---|---|
| Viewer | Просмотр dashboard, фильтрация, просмотр карточек SKU и камер |
| Analyst | Все права Viewer, плюс доступ к аналитическим срезам и выгрузкам |
| Admin | Все права Analyst, плюс редактирование справочников, импорт XLSX, управление пользователями и аудитом |

### 3.2 Основные сценарии

- Руководитель склада открывает главный dashboard и видит текущую заполненность склада, свободные паллетоместа и проблемные камеры.[cite:2][cite:5]
- Аналитик выбирает период, SKU и камеру хранения, чтобы изучить динамику занятости и отгрузок.[cite:3][cite:8]
- Администратор загружает новый XLSX-файл из 1С, проверяет preview, исправляет ошибки маппинга и запускает импорт в базу.[cite:12]
- Пользователь открывает карточку SKU и анализирует, сколько ячеек SKU занимает, как долго хранится и насколько эффективно отгружается относительно объема хранения.[cite:13]

## 5. Функциональные требования

### 5.1 Главный dashboard

Главный экран должен показывать текущее состояние склада и динамику за выбранный период. Пользователь должен получать ответ на три вопроса: сколько мест свободно, чем занят склад, какие SKU используют площадь эффективно или неэффективно.[cite:2][cite:3][cite:13]

Обязательные элементы главного dashboard:

- KPI-карточки;
- глобальные фильтры;
- график заполненности по камерам;
- график динамики занятости по датам;
- аналитика по SKU;
- таблица детализации по камерам / SKU / датам;
- индикатор свежести данных (последний импорт, дата и время).

### 5.2 Раздел «Камеры хранения»

Раздел должен позволять создавать, редактировать и архивировать камеры хранения. Для каждой камеры требуется хранить код, наименование, тип, емкость, статус активности, число паллетомест и дополнительные признаки (например, температурный режим, зона хранения, складской блок).

Внутри камеры должна быть возможность вести состав паллетомест. Для паллетоместа требуется фиксировать идентификатор, принадлежность к камере, статус (свободно, занято, заблокировано, недоступно), дату последнего изменения и примечание.

### 5.3 Раздел «Справочник SKU»

Справочник SKU должен содержать:

- код SKU;
- наименование;
- категорию;
- единицу измерения;
- коэффициент паллетизации;
- габаритные или объемные характеристики (если доступны);
- статус активности;
- служебные поля аудита.

SKU должны использоваться как мастер-справочник для импорта фактических данных. При загрузке XLSX новые или неизвестные SKU должны попадать в журнал ошибок либо в отдельный режим согласования, в зависимости от выбранной политики загрузки.

### 5.4 Импорт XLSX из 1С

Импорт должен быть реализован как ETL-конвейер с промежуточным staging-слоем, preview и журналированием. Это снижает риск повреждения боевых данных и позволяет валидировать файл до фиксации изменений в основной базе.[cite:12]

Требования к импорту:

- загрузка одного или нескольких XLSX-файлов;
- выбор типа шаблона импорта;
- маппинг колонок файла на поля системы;
- предварительный просмотр строк;
- валидация типов данных, дат, числовых значений, обязательных полей;
- проверка существования SKU и камер хранения;
- дедупликация строк;
- upsert в основную БД;
- логирование результата загрузки;
- отображение ошибок по строкам.

Статусы импорта:

| Статус | Описание |
|---|---|
| Uploaded | Файл загружен, еще не разобран |
| Parsed | Файл успешно прочитан |
| Validated | Валидация завершена |
| Warning | Есть предупреждения, импорт допустим |
| Error | Есть критические ошибки, импорт запрещен |
| Imported | Данные записаны в БД |
| Rolled Back | Импорт отменен или откатан |

### 5.5 Аналитика по SKU

Раздел аналитики SKU должен показывать:

- текущее количество занятых паллетомест по SKU;
- среднее количество занятых паллетомест за период;
- максимальное количество занятых паллетомест за период;
- суммарный объем хранения за период;
- объем отгрузок за период;
- коэффициент эффективности хранения;
- рейтинг SKU по эффективности / неэффективности.[cite:3][cite:13]

### 5.6 Аудит и журнал действий

Необходимо вести аудит:

- входов в систему;
- изменения справочников;
- запусков импорта;
- ошибок импорта;
- ручных корректировок данных;
- изменений прав пользователей.

## 6. Нефункциональные требования

- Интерфейс desktop-first, адаптивный для экранов от 1280 px и корректный на планшетах.[cite:4]
- Поддержка светлой и темной темы интерфейса для удобства длительной работы в аналитическом приложении.[cite:4]
- Основной сценарий использования — работа во внутренней корпоративной сети.
- Все критические операции должны быть журналируемыми.
- Система должна поддерживать ролевую модель доступа.
- Время открытия главного dashboard при штатном объеме агрегированных данных — не более 3 секунд.
- Загрузка файла XLSX до 20 МБ — не более 60 секунд до получения результата валидации.
- Должно поддерживаться безопасное восстановление после ошибок импорта.

## 7. Логическая модель данных

### 7.1 Основные сущности

| Сущность | Назначение |
|---|---|
| Warehouse | Склад или логическая площадка хранения |
| Chamber | Камера хранения |
| PalletLocation | Паллетоместо / ячейка |
| SKU | Номенклатурная позиция |
| StockSnapshot | Снимок состояния хранения на дату |
| ShipmentFact | Факт отгрузки SKU |
| MovementFact | Перемещение / размещение / освобождение |

| User | Пользователь системы |
| AuditLog | Журнал действий |

### 7.2 Связи сущностей

- Один склад содержит много камер хранения.
- Одна камера содержит много паллетомест.
- Один SKU может одновременно занимать много паллетомест.
- StockSnapshot хранит состояние SKU в разрезе даты, камеры и/или паллетоместа.
- ShipmentFact хранит отгрузки SKU по датам.
- ImportBatch содержит много ImportRow.
- AuditLog привязан к пользователю и объекту действия.

## 9. Источники данных и правила загрузки

### 9.1 Предполагаемые типы XLSX из 1С

Система должна поддерживать минимум три шаблона:

| Шаблон | Назначение |
|---|---|
| Остатки на дату | Состояние занятости склада и распределение SKU |
| Отгрузки за период | Фактические отгрузки по SKU |
| Операции / движения | Размещения, перемещения, освобождения |

### 9.2 Обязательные поля для импорта остатков

- дата снимка;
- склад;
- камера хранения;
- паллетоместо или число занятых мест;
- код SKU;
- количество;
- объем, если доступен.

### 9.3 Обязательные поля для импорта отгрузок

- дата отгрузки;
- склад;
- код SKU;
- количество отгружено;
- объем отгружено, если доступен.

### 9.4 Правила валидации

- дата обязательна и должна быть корректной;
- SKU должен существовать в справочнике или быть явно разрешен для авто-создания;
- камера должна существовать в справочнике;
- числовые поля должны быть неотрицательными;
- дубли по natural key должны быть выявлены до записи в основной слой;
- строки с критическими ошибками не должны импортироваться.[cite:12]

### 9.5 Natural key для дедупликации

Рекомендуемые natural key:

- для остатков: `snapshot_date + warehouse + chamber + pallet_location + sku`;
- для отгрузок: `shipment_date + warehouse + sku + document_number`;
- для движений: `operation_datetime + warehouse + chamber + pallet_location + sku + operation_type`.

## 10. Формулы KPI и аналитических показателей

### 10.1 Базовые показатели

1. **Всего паллетомест**  
   Сумма всех активных паллетомест в активных камерах.

2. **Свободные паллетоместа**  
   Активные паллетоместа со статусом `free` на текущую дату или по последнему доступному snapshot.

3. **Занятые паллетоместа**  
   Активные паллетоместа со статусом `occupied`.

4. **Процент заполненности склада**  
   \[
   occupancy\_rate = \frac{occupied\_locations}{total\_active\_locations} \times 100
   \]

5. **Процент свободных мест**  
   \[
   free\_rate = \frac{free\_locations}{total\_active\_locations} \times 100
   \]

### 10.2 Показатели по камерам

- Число паллетомест в камере.
- Число свободных мест в камере.
- Число занятых мест в камере.
- Процент заполненности камеры.
- Средняя заполненность камеры за период.
- Максимальная заполненность камеры за период.

Такие метрики непосредственно связаны с warehouse utilization и location usage, которые обычно используются в складской аналитике для оценки загрузки мощностей.[cite:3][cite:7][cite:13]

### 10.3 Показатели по SKU

- Текущее количество занятых паллетомест SKU.
- Среднее количество занятых паллетомест SKU за период.
- Максимум занятых паллетомест SKU за период.
- Количество дней хранения в периоде.
- Суммарный объем хранения SKU за период.
- Суммарный объем отгрузок SKU за период.
- Коэффициент оборачиваемости SKU.
- Коэффициент эффективности хранения SKU.[cite:3][cite:13]

### 10.4 Рекомендуемые формулы эффективности

#### Вариант A: коэффициент оборачиваемости хранения

\[
turnover\_efficiency = \frac{shipped\_volume\_period}{avg\_occupied\_volume\_period}
\]

Чем выше показатель, тем лучше SKU использует складскую емкость по отношению к отгрузкам.

#### Вариант B: коэффициент складской нагрузки

\[
storage\_load\_ratio = \frac{avg\_occupied\_locations\_period}{shipped\_qty\_period}
\]

Чем выше значение, тем больше складских мест требуется на единицу отгрузки, то есть тем ниже эффективность SKU.

#### Вариант C: интегральный score для ранжирования

\[
sku\_efficiency\_score = w1 \times norm(turnover) - w2 \times norm(avg\_occupied\_locations) - w3 \times norm(storage\_days)
\]

Итоговая формула должна быть утверждена заказчиком после валидации на исторических данных, так как разные компании трактуют «эффективность SKU» по-разному.[cite:2][cite:3]

- Клик по значению KPI → переход в таблицу с предфильтрацией.
- Выбор периода → пересчет всех KPI, графиков и таблиц.

## 14. Правила расчета и агрегации

- Текущие KPI должны считаться по последнему доступному корректному snapshot.
- Периодная аналитика должна строиться по выбранному диапазону дат.
- Для отсутствующих дат возможны два режима: хранить только фактические снимки или достраивать календарную серию; режим должен быть утвержден до разработки.
- Если в один день загружено несколько файлов одного типа, приоритет определяется временем импорта и правилом актуальности batch.
- Все расчеты в dashboard должны выполняться из проверенного аналитического слоя, а не напрямую из raw import-rows.

## 15. ETL-процесс

### 15.1 Этапы ETL

1. Upload файла.
2. Parse листов XLSX.
3. Mapping колонок.
4. Validation.
5. Preview результата.
6. Commit в staging.
7. Upsert в production tables.
8. Rebuild агрегатов.
9. Логирование результата.

### 15.2 Обработка ошибок

Ошибки делятся на два уровня:

- критические: неверный формат файла, отсутствие ключевых колонок, некорректный тип даты, нарушение ссылочной целостности;
- некритические: неизвестная категория, пустое необязательное поле, предупреждения по округлению.

При критических ошибках commit запрещается. При некритических предупреждениях пользователь уровня Admin может выполнить импорт после подтверждения.[cite:12]

## 16. Безопасность и доступ

- Ролевой доступ на уровне API и UI.
- JWT или server-session авторизация в зависимости от внутренней инфраструктуры.
- Ограничение доступа к импорту и справочникам только для роли Admin.
- Логирование авторизаций и административных операций.
- Валидация загружаемых файлов по MIME, расширению и размеру.
- Защита от повторной загрузки одинаковых файлов через checksum.

## 18. Требования к frontend

- React SPA или hybrid web app.
- Sidebar navigation.
- Единая зона основного скролла.
- Таблицы с сортировкой, фильтрацией и sticky header.
- Графики должны поддерживать hover, tooltip, legend, zoom при необходимости.
- Числовые значения отображать в tabular-nums.
- Все KPI пересчитываются при изменении фильтров.
- Обязательные состояния: loading, empty, error, stale data.[cite:4]

## 19. Требования к backend

- Слой API отделен от ETL и от расчетного слоя.
- Импорт XLSX не должен блокировать остальные пользовательские запросы.
- Поддержка batch-processing и повторного запуска неуспешной валидации.
- Вычисление тяжелых агрегатов возможно через materialized views или nightly refresh.
- Все бизнес-формулы должны быть вынесены в отдельный сервисный слой.

## 20. Отчетность и экспорт

На первом этапе экспорт не обязателен, но архитектура должна позволять в будущем добавить:

- выгрузку таблиц в XLSX / CSV;
- печатную форму dashboard;
- выгрузку журнала импорта;
- автоматическую рассылку KPI по email.

## 21. Критерии приемки

Система считается принятой, если выполнены условия:

1. Пользователь может открыть dashboard и увидеть текущие KPI склада.
2. Система показывает свободные и занятые паллетоместа по камерам.
3. Система показывает занятость SKU в текущем состоянии и за выбранный период.
4. Система показывает аналитический показатель эффективности SKU на основе хранения и отгрузок.[cite:2][cite:3][cite:13]
5. Администратор может загрузить XLSX из 1С, пройти preview и записать данные в БД.[cite:12]
6. Ошибки импорта отображаются построчно.
7. Справочники камер и SKU редактируются через UI.
8. Работают ролевые права доступа.
9. Все основные действия фиксируются в журнале аудита.
10. Dashboard корректно работает на актуальных версиях Chrome и Edge.

## 25. Риски проекта

- неоднородные XLSX-шаблоны из 1С;
